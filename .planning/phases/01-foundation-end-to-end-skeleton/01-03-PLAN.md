<objective>
Build the interactive 2-step assessment wizard (Registration + Company
Profile) as a client component with local step/answers state, a persistent
brand-styled progress bar, and 200ms step transitions — the user-facing half
of the walking skeleton.

Purpose: This is the only user-visible surface built in this wave; it must
be fully navigable and validated on its own so Plan 01-04 only has to wire
one `onSubmit` function into it, not build any new UI.
Output: A working `/assessment` route where a user can complete both Phase 1
steps with no data loss going back and forward. The final Submit action
calls an `onSubmit` prop whose real network implementation is supplied by
Plan 01-04 (which also builds the API it calls) — this plan defines that
prop's contract and renders the button that calls it.
</objective>

<threat_model>
## Trust Boundaries

None new server-side — this plan is entirely client-side state and form
rendering. The real trust boundary (untrusted browser input crossing into
the server) is introduced by the API route in Plan 01-04, which performs
the authoritative validation.

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Tampering | Client-side validation in `StepRegistration`/`StepCompanyProfile` | accept | Client validation here is a UX affordance only, not a security boundary; a user could bypass it via devtools, but the server-side handler built in Plan 01-04 re-validates every field before persisting, so no untrusted data reaches the database on the strength of client validation alone |
</threat_model>
