  <behavior>
    - `POST` with a body missing `contactName` (or any required field) returns
      status 400 and `{ error: string }` naming the missing field; no
      `prisma.assessment.create` call is made.
    - `POST` with `companyProfile.industry` not present in `INDUSTRY_OPTIONS`
      returns 400 (same for `employeeBand`/`EMPLOYEE_BANDS`,
      `revenueBand`/`REVENUE_BANDS`, `businessModel`/`BUSINESS_MODEL_OPTIONS`,
      and any `leadershipGoals` entry not present in `LEADERSHIP_GOALS`).
    - `POST` with a fully valid `WizardAnswers` body returns status 201 and
      `{ id: string }`, and a subsequent `GET /api/assessments/{id}` returns
      200 with a JSON body whose `totalScore` and `stage` match what
      `calculateTotalScore`/`getStage` would compute for that same input.
    - `GET` for a non-existent id returns 404 and `{ error: string }`.
  </behavior>
  <action>
    Create `app/api/assessments/route.ts` exporting an async `POST(request: Request)`
    App-Router route handler. Parse the JSON body as `WizardAnswers`. Validate
    manually (no schema library, per the design decision recorded in
    `01-SKELETON.md`): `contact.contactName`, `contact.contactEmail`,
    `contact.companyName` are non-empty strings;
    `contact.contactEmail` matches `/^[^\s@]+@[^\s@]+\.[^\s@]+$/`;
    `companyProfile.industry` is a member of `INDUSTRY_OPTIONS`;
    `companyProfile.employeeBand` is a member of `EMPLOYEE_BANDS`;
    `companyProfile.revenueBand` is a member of `REVENUE_BANDS`;
    `companyProfile.businessModel` is a member of `BUSINESS_MODEL_OPTIONS`;
    every entry of `companyProfile.leadershipGoals` is a member of
    `LEADERSHIP_GOALS`; `companyProfile.blendedHourlyCost` is a finite number
    &gt; 0. On the first failed check, return
    `NextResponse.json({ error: '&lt;specific field&gt; is invalid' }, { status: 400 })`
    and stop (do not continue validating further fields). On success, build
    a `ScoringInput` of `{ companyProfile: body.companyProfile }` (all other
    sections omitted/undefined since Phase 1 does not collect them), call
    `calculateCategoryScores`, then `calculateTotalScore`, then `getStage`.
    Call `prisma.assessment.create({ data: { contactName: body.contact.contactName, contactEmail: body.contact.contactEmail, contactPhone: body.contact.contactPhone || null, companyName: body.contact.companyName, companyProfile: body.companyProfile, departments: Prisma.JsonNull, techStack: Prisma.JsonNull, dataReadiness: Prisma.JsonNull, aiAdoption: Prisma.JsonNull, automation: Prisma.JsonNull, agentInterest: Prisma.JsonNull, security: Prisma.JsonNull, employeeReadiness: Prisma.JsonNull, categoryScores, totalScore, stage: stage.label, dataCompleteness: null, benchmark: Prisma.JsonNull, swot: Prisma.JsonNull, opportunityMatrix: Prisma.JsonNull, risk: Prisma.JsonNull, roadmap: Prisma.JsonNull, roi: Prisma.JsonNull, executiveSummary: null } })`
    (import `Prisma` as a named import from `@prisma/client` for
    `Prisma.JsonNull`). Return
    `NextResponse.json({ id: created.id }, { status: 201 })`. Wrap the
    Prisma call in try/catch; on error return
    `NextResponse.json({ error: 'Failed to save assessment' }, { status: 500 })`.
    Create `app/api/assessments/[id]/route.ts` exporting an async
    `GET(request: Request, { params }: { params: { id: string } })`:
    `prisma.assessment.findUnique({ where: { id: params.id } })`; if `null`,
    return `NextResponse.json({ error: 'Not found' }, { status: 404 })`;
    otherwise return `NextResponse.json(assessment, { status: 200 })`.
  </action>

  </verify>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser -&gt; `POST /api/assessments` | Untrusted user input from the wizard crosses into the server and is persisted to SQLite |
| Anyone with a `/results/[id]` URL -&gt; that record | No authentication gates read access to any assessment |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Tampering | `POST /api/assessments` body | mitigate | Manual server-side validation (field presence, email format, enum membership against the shared option-list constants, positive-number check) rejects any malformed or out-of-range payload with 400 before any data reaches Prisma |
| T-04-02 | Injection | Prisma queries in both route handlers | accept | Prisma's query builder parameterizes all values; no raw SQL is used anywhere in this plan |
| T-04-03 | Information Disclosure | `GET /api/assessments/[id]` and `/results/[id]` have no auth | accept | Consistent with the project's locked "no authentication" constraint (single-user, local-only tool for v1); `id` is a `cuid()`, effectively unguessable, and the app is not exposed to the public internet in Phase 1 |
| T-04-04 | Denial of Service | No rate limiting on `POST /api/assessments` | accept | Local-only dev tool (`npm run dev`), not internet-exposed in this phase; revisit if/when a hosted deployment is introduced (explicitly out of scope for all of v1 per PROJECT.md constraints) |
| T-04-05 | Repudiation / Spoofing / Elevation of Privilege | N/A | accept | No user accounts or privilege levels exist anywhere in this application |
</threat_model>
